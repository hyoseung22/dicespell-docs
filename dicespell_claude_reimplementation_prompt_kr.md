# Claude 전달용 프롬프트 초안

아래 프롬프트는 Claude에게 현재 DiceSpell Standalone 빌드를 최대한 동일하게 분석하거나 재구현하라고 요청할 때 바로 붙여넣기 위한 초안이다.

---

다음 두 문서를 기준으로 `DiceSpell Standalone`의 현재 빌드를 완전히 이해하고, 필요한 경우 동일하게 재구현할 수 있도록 분석해줘.

- `dicespell_current_build_reverse_spec_kr.md`
- `dicespell_current_build_code_map_kr.md`

요청 사항:

1. 두 문서를 읽고 현재 게임의 전체 시스템을 빠짐없이 이해해줘.
2. 문서와 코드 구조를 바탕으로 현재 빌드와 동일한 기능을 재구현하기 위한 개발 계획을 작성해줘.
3. 시스템별로 다음 항목을 분리해서 정리해줘.
   - 로비
   - 책 선택
   - 전투
   - 몬스터 AI
   - 붕괴 시스템
   - 상단 합계 보너스
   - 보상
   - 상점
   - 아티팩트
   - 정령석/강화석
   - 저장/복구
   - 오디오
   - 해상도 옵션
   - 승률 시뮬레이터
4. 각 시스템마다 아래 항목을 함께 써줘.
   - 데이터 구조
   - 상태 전이
   - UI 요구사항
   - 실제 수치/확률/공식
   - 구현 시 주의점
5. 원본 코드에 죽은 코드와 문자열 인코딩 문제가 있으므로, 문서 기준으로 정제된 사양을 우선 사용해줘.
6. “추정”과 “명시된 사실”을 구분해서 적어줘.
7. 최종 결과는 실제 개발자가 그대로 작업에 들어갈 수 있는 수준의 상세 구현 문서로 작성해줘.

추가 지시:

- 문서에 있는 수치, 확률, ID, 페이지 규칙, 아티팩트 효과를 임의로 단순화하지 마.
- 범위를 축소하지 말고 현재 빌드의 모든 시스템을 포함해줘.
- 가능하면 재구현 순서와 파일 구조 제안도 함께 작성해줘.
- 소스코드에 보이는 깨진 문자열은 그대로 복사하지 말고 문맥상 정상 한국어로 정리해줘.

최종 산출물 형식:

1. 현재 빌드 시스템 요약
2. 시스템별 상세 사양
3. 상태 스키마 정리
4. 구현 순서 제안
5. 리스크와 애매한 지점
6. 재구현 체크리스트

---

필요하면 이 프롬프트 뒤에 실제 소스코드 파일을 추가 컨텍스트로 붙여도 된다. 다만 우선순위는 문서 두 개를 더 높게 둬야 한다.
