# DiceSpell 첫 런 온보딩 handoff packet

## 빠른 안내

### 3줄 요약
- 이 문서는 첫 런 10분 안에 무엇을 어떤 순서로 읽히게 만들지 정하는 **온보딩 원본 handoff**다.
- 프로그래머는 `surface / 상태 / 주입 위치`를 먼저 보고, UI·아트는 `메시지 밀도 / 강조 방식`, QA는 `반복 노출 금지와 오해 방지`를 먼저 보면 된다.
- 상세 카피나 검수표가 필요하면 뒤의 `screen / acceptance / content deck`으로 내려가고, 이 문서는 **범위와 역할 경계**를 고정하는 기준으로 읽는다.

**한 줄 목표:** 첫 런 플레이어가 `책 선택 이유 → 첫 턴 규칙 → reward/shop 차이 → ending/overrun 의미`를 화면 순서대로 배우게 만든다.

### 왜 이 문서를 먼저 보나
- 이 문서는 `왜 필요한가`, `무엇을 일부러 하지 않는가`, `어떤 surface를 먼저 붙이는가`를 가장 먼저 고정한다.
- 노션/컨플에 그대로 붙여 넣어도 범위가 흔들리지 않게, 설명보다 **실행 가능한 기준 문장**을 우선 유지한다.

### 누가 어디부터 읽으면 되나
| 역할 | 먼저 볼 부분 | 바로 이어서 볼 문서 |
| --- | --- | --- |
| 프로그래머 | 포함/제외 범위, 4개 surface, 상태 저장 기준 | `screen packet`, `integration workorder` |
| UI | 4개 surface, 메시지 밀도, bounded 표현 원칙 | `screen packet`, `content deck` |
| 아트 | 강조가 필요한 순간, asset 최소화 기준 | `screen packet`, `production cutsheet` |
| QA/PM | 왜 필요한가, 반복 노출 금지, 오해 포인트 | `acceptance matrix`, `summary` |

## 문서 목적

이 문서는 현재 DiceSpell 스탠드얼론 빌드가 이미 갖춘 `전투 규칙 / 책 정체성 / 보상-오버런 연결감`을 **처음 들어온 플레이어가 실제로 읽고 따라갈 수 있게 만드는 bounded 온보딩 작업**의 기준을 고정한다.

지금까지 문서와 구현은 주로 아래 축을 밀어 왔다.

- `한 턴 = 한 주문` 전투 규칙 고정
- 책별 정체성/고유 규칙/완주 보너스 차별화
- late normal 보상 → 적 처치 전리품 → 책 클리어 `오버런 전리품 보관함` → `overrunEvent` handoff까지 연결
- 오버런 경계 이벤트의 프로그래머 / UI / 아트 / QA handoff 정리

하지만 첫 런 관점에서는 아직 큰 빈칸이 남아 있다.

- 어떤 책을 골라야 하는지 정보는 늘었지만, **처음 무엇을 읽고 무엇을 나중에 읽어도 되는지** 우선순위가 부족하다.
- 전투 규칙은 이미 날카롭게 정리됐지만, **`이번 턴에 뭘 하면 되는가`와 `왜 손해를 봤는가`를 첫 2~3전투 안에 익히는 장치**가 약하다.
- reward / shop / ending / overrun의 의미는 문서상 분명해졌지만, **실제 빌드 안에서는 초보자에게 한 번에 너무 많은 시스템이 열린다.**

이 문서의 결론은 간단하다.

> 다음 온보딩 작업은 `대형 튜토리얼 모드`가 아니라, **첫 런 10분 안에 필요한 정보만 짧게 끊어 보여 주는 4개 surface의 bounded guidance pass**여야 한다.

즉 이번 handoff는 시스템을 더 설명형으로 늘리는 것이 아니라, **이미 좋은 시스템이 초반 이탈 없이 읽히도록 최소한의 레일을 놓는 작업**이다.

## 1. 왜 지금 이 문서가 필요한가

현재 tracker 기준으로 오버런 경계 이벤트(`overrunEvent`) 문서 묶음은 이미 구현 착수 가능한 수준까지 좁혀졌다. 즉 지금 남은 오버런 축의 blocker는 더 이상의 handoff 공백이 아니라 **실제 코드 컷오버**다.

반면 다음으로 크게 남은 문서 공백은 `튜토리얼/온보딩`이다.

현재 상태를 초보자 입장에서 다시 읽으면 문제가 선명하다.

1. **책 선택 화면의 정보량은 많지만 읽기 순서가 없다.**  
   `subtitle / mood / playstyle / openingPlan / caution / tag / signature rule / completion bonus`가 모두 의미는 있지만, 초보자는 `지금 무엇을 기준으로 책을 고르면 되는가`를 빠르게 못 잡을 수 있다.

2. **전투의 핵심 규칙은 강하지만 첫 해석 비용이 높다.**  
   `한 턴 = 한 주문`, `족보 잠금`, `상단 합계`, `MP 부족 시 자동 종료`, `붕괴 압박`은 모두 중요한데, 이를 같은 전투 화면에서 처음 만나면 우선순위가 뒤섞인다.

3. **보상과 상점의 역할 차이가 초반에 명시적으로 배우기 어렵다.**  
   이 게임은 이미 `전투 보상 = 단기 전술`, `상점 = 장기 빌드`로 잘 갈라져 있지만, 처음 하는 플레이어는 둘 다 그냥 `카드 고르는 화면`으로 읽을 수 있다.

4. **책 클리어/실패/오버런의 메타 의미가 한 번에 들어오지 않는다.**  
   `서고 기록`, `완주 보너스`, `오버런`, `이번 세그먼트 전리품`은 시스템적으로 연결돼 있지만, 초보자는 `이겼는데 왜 아직 안 끝났지?`, `졌는데 무엇이 남았지?`에서 잠깐 흔들릴 수 있다.

즉 지금 필요한 것은 방대한 설명서가 아니라, **초보자가 첫 10분 안에 `무엇부터 이해하면 되는가`를 화면 자체에서 배우는 최소한의 가이드 구조**다.

## 2. 이번 bounded 온보딩의 한 줄 목표

> 플레이어가 첫 런 10분 안에 `책 선택 이유`, `전투 1턴의 구조`, `보상/상점의 차이`, `런 종료 후 남는 것`을 각각 한 번씩만 정확히 배우게 만든다.

이 목표가 지켜지면 충분하다.

- 모든 시스템을 첫 화면에서 설명하지 않는다.
- 첫 런에서 한 번 본 뒤 반복 노출을 최소화한다.
- 게임 템포를 끊지 않는다.
- 이미 존재하는 UI 패널과 카피를 최대한 재사용한다.
- 새 규칙을 추가하지 않고, 기존 규칙을 **올바른 순서로 보이게만** 만든다.

## 3. 범위 정의

### 포함

- 첫 런 전용 또는 첫 도달 전용 guidance surface 정의
- 책 선택 / 첫 전투 / 첫 reward·shop / 첫 ending·overrun 4구간 온보딩 구조 정의
- 프로그래머 / UI / 아트 / QA handoff 정의
- 카피 우선순위와 금지 패턴 정의
- 최소 상태 저장(`seen` 플래그류) 가이드 정의
- smoke / 수동 QA 기준 정의

### 제외

- 별도 튜토리얼 전투 맵 제작
- 샘플 덱/고정 적/가짜 전투 시나리오 추가
- 장문의 codex/백과사전형 도움말 창 제작
- 신규 메타 화폐/튜토리얼 보상 추가
- 음성/컷신/대형 삽화 파이프라인 전제
- 접근성/로컬라이징 전체 정비

즉 이번 작업은 `튜토리얼 모드`가 아니라 **실서비스형 첫 런 guidance layer**다.

## 4. 현재 빌드 기준 실제 문제를 어디서 푸는가

### 4.1 도서관(책 선택)에서 풀 문제

플레이어는 첫 선택에서 다음 네 가지를 헷갈릴 수 있다.

- `추천 운영`과 `주의점` 중 무엇을 먼저 읽어야 하는가
- 책의 정체성이 `지금 강한 수치`인지 `운영 습관`인지
- 첫 책을 무엇으로 고르면 무난한가
- `완주 보너스`가 지금 의미 있는 정보인지, 나중 정보인지

따라서 여기서는 **책 카드 정보를 전부 늘리는 대신, 읽는 순서를 가르쳐야 한다.**

### 4.2 첫 전투에서 풀 문제

전투 첫 2~3턴 동안 플레이어는 다음을 동시에 배운다.

- 주사위를 굴리고 고른다.
- 쓸 수 있는 족보를 본다.
- 한 턴에 주문이 한 번만 나간다.
- 쓰면 잠긴다.
- MP가 부족하면 턴이 바로 넘어간다.
- 상단 합계와 붕괴가 장기 압박으로 남는다.

하지만 초반에 진짜 필요한 건 이 전부가 아니다. 우선순위는 아래 세 가지다.

1. **이번 턴에 주문은 한 번만 쓴다.**
2. **쓸 수 있는 족보를 먼저 본다.**
3. **왜 이 주문이 나갔고 왜 다음 족보가 잠겼는지 로그와 카드에서 읽는다.**

즉 첫 전투 guidance는 `시스템 전체 설명`이 아니라 **행동 순서와 손익 해석**을 도와야 한다.

### 4.3 첫 reward / shop에서 풀 문제

현재 구조는 이미 좋다.

- reward는 즉시 적용 중심
- shop은 장기 빌드 투자 중심

문제는 첫 플레이어가 이 차이를 즉시 안다고 가정하면 안 된다는 점이다. 따라서 여기서는 신규 설명창보다 **한 줄 목적 문구**가 더 중요하다.

### 4.4 첫 ending / overrun에서 풀 문제

현재 빌드는 실패해도 `서고 기록`이 남고, 책을 클리어하면 완주 보너스와 오버런이 이어진다. 하지만 초보자는 여기서 아래를 헷갈릴 수 있다.

- 패배해도 무엇이 남는가
- 책을 끝내면 이번 런이 끝난 것인지 이어지는 것인지
- 오버런이 강제인지 선택인지
- 지금 이 화면이 정산인지, 다음 책 준비인지

따라서 ending guidance는 승리/패배를 다 길게 설명하는 대신, **`이번 결과로 남는 것`과 `지금 누르는 버튼의 의미`를 짧게 고정**해야 한다.

## 5. 제안 구조: 4개 surface의 bounded guidance pass

이번 온보딩은 별도 모드가 아니라, 아래 4개 surface에 분산된 짧은 안내로 설계한다.

1. `도서관 책 선택 primer`
2. `첫 전투 primer`
3. `첫 reward / 첫 shop primer`
4. `첫 ending / 첫 overrun primer`

핵심 원칙은 아래와 같다.

- 각 primer는 **한 화면 한 메시지**만 준다.
- modal을 남발하지 않고, 기존 패널 위 `hero-card` / `callout` / `inline badge` 구조를 우선 재사용한다.
- 닫기/확인 후 같은 run 안에서는 반복하지 않는다.
- 이미 익힌 플레이어를 방해하지 않도록 `다시 보지 않기` 또는 `이번 런에서는 숨기기`가 가능해야 한다.

## 6. surface별 상세 spec

## 6.1 도서관 책 선택 primer

### 목표

플레이어가 `책을 무엇으로 고를지`보다 먼저, **책 카드에서 무엇을 먼저 읽어야 하는지**를 배운다.

### 노출 시점

- 메타 진행도가 사실상 초기 상태인 첫 런
- 또는 `첫 책 선택 진입` 1회

### 핵심 메시지

> 첫 책은 `추천 운영`과 `주의점`만 먼저 읽어도 충분하다. 완주 보너스는 나중 보상이다.

### 화면 구성

- 책 선택 패널 상단 또는 우측에 `짧은 primer 카드` 1개 노출
- 카드 안에는 다음 3줄만 둔다.
  1. `추천 운영`: 이 책으로 어떤 플레이를 권하는가
  2. `주의점`: 초반에 가장 자주 망하는 실수
  3. `완주 보너스`: 지금 외울 필요 없이, 책 성격만 읽으면 된다

### UI 규칙

- 책 카드 전체를 덮는 fullscreen modal은 금지
- primer 카드에서 실제 책 카드의 `openingPlan` / `caution` / `signatureRule` 위치를 얇은 highlight로 짚는 것은 허용
- `추천 시작` 뱃지는 가장 무난한 1~2권에만 붙이고, 전체 책에 강/약 서열처럼 보이게 붙이지 않는다

### 카피 톤

- `이 책이 가장 쉽습니다`보다 `첫 런이면 이런 운영이 읽기 쉽다`가 낫다
- 수치 우열보다 운영 성향을 먼저 말한다
- 초보자에게 겁을 주는 표현(`실패하기 쉽다`, `고난도라 하지 마세요`)은 피하고, 대신 `이 책은 이런 실수가 잘 난다`처럼 구체적으로 쓴다

### 프로그래머 handoff

- `meta` 또는 `uiPrefs`에 `seenBookSelectPrimer` 같은 불리언 저장
- primer 노출 조건은 `first-run` 또는 `manual help reopen` 두 축으로 관리
- 책 데이터에 새 강제 필드를 늘리지 말고, 기존 `playstyle/openingPlan/caution`를 재조합해 쓴다

### UI handoff

- primer 카드는 책 목록을 가리지 않는 우측/상단 보조 패널 우선
- 모바일 대응이 지금 범위는 아니지만, 최소한 1280 폭에서 레이아웃 붕괴가 없어야 한다

### 아트 handoff

- 기존 `library` 카드 톤과 맞는 소형 리본/배지 1종이면 충분
- 책별 전용 새 삽화는 범위 밖

## 6.2 첫 전투 primer

### 목표

플레이어가 첫 전투에서 **이번 턴 행동 순서**와 **실패 원인 읽기**를 배운다.

### 노출 시점

- run 기준 첫 전투 진입 직후 1회
- 필요 시 첫 주문 사용 후 1회 후속 힌트

### 1차 메시지

> 매 턴 핵심은 하나다. `굴리고 → 가능한 족보를 보고 → 주문 1회`.

### 2차 메시지

> 주문을 쓰면 해당 족보는 이 페이지 동안 잠길 수 있다. 지금 중요한 건 `한 번의 좋은 선택`이다.

### 화면 구성

#### A. 전투 진입 직후 primer card

- 중앙 또는 우측 로그 위에 compact hero-card 1개
- 아래 세 줄만 보여 준다.
  - `1. 가능한 족보를 먼저 본다`
  - `2. 이번 턴 주문은 한 번만 쓴다`
  - `3. 못 쓰면 턴이 자동으로 넘어간다`

#### B. 첫 주문 사용 직후 inline hint

- 최근 로그 또는 hand panel 상단에 짧은 inline hint
- 메시지 예시:
  - `방금 사용한 족보는 이번 페이지 동안 잠길 수 있습니다.`
  - `로그에서 피해/상태이상/잠금 결과를 바로 확인할 수 있습니다.`

#### C. 첫 MP 부족 자동 종료 직후 hint

- 별도 modal 금지
- 기존 로그 옆 짧은 강조 텍스트로 처리
- 메시지 예시:
  - `지금은 쓸 수 있는 주문이 없어 몬스터 턴으로 넘어갔습니다.`

### 우선순위

첫 전투 primer는 아래 3개만 확실히 읽히면 된다.

1. 주문은 턴당 1회
2. 가능한 족보를 먼저 보기
3. 로그/잠금 결과 읽기

`상단 합계`, `붕괴`, `속성`, `상태이상`은 첫 전투 진입에서 다 설명하지 않는다. 그 정보는 **패널 제목/뱃지/로그**로만 약하게 노출하고, 본격 튜토리얼 설명은 하지 않는다.

### 프로그래머 handoff

- `seenBattlePrimer`, `seenFirstSpellHint`, `seenAutoPassHint` 정도의 최소 플래그만 둔다
- 첫 주문 감지는 기존 액션 처리 후킹으로 해결하고 별도 전투 모드 분기는 만들지 않는다
- primer 노출이 상태 전이에 개입하지 않게 한다. 즉 `battle` phase 안의 UI 보조 상태여야 한다

### UI handoff

- primer가 몬스터/주사위/족보 핵심 인터랙션을 가리면 안 된다
- `다시 보지 않기`보다 `알겠음` + `도움말 다시 보기`가 낫다
- 로그 위 compact 카드 또는 hand panel 상단 리본 구조를 우선 검토

### 아트 handoff

- 새 아이콘은 `주사위`, `주문`, `잠금` 3종의 공용 glyph 정도면 충분
- 전투용 풀 일러스트는 범위 밖

---

## 6.3 첫 reward / 첫 shop primer

### 목표

플레이어가 reward와 shop을 둘 다 `고르는 화면`으로 보지 않고, **역할이 다른 두 공간**으로 인식하게 만든다.

### 노출 시점

- 첫 reward 진입 시 1회
- 첫 shop 진입 시 1회

### reward 핵심 메시지

> 보상은 다음 1~2전투를 버티고 밀어 주는 즉시 선택이다.

### shop 핵심 메시지

> 상점은 이번 런의 장기 방향을 굳히는 투자 공간이다.

### 화면 구성

- reward / shop 헤더 아래 한 줄 purpose line 고정
- 첫 진입일 때만 해당 문구가 accent badge + 1문장으로 조금 더 강조
- 설명은 각 1문장으로 제한

### 금지 사항

- reward와 shop의 모든 하위 타입(정령석/강화석/유물/회복)을 첫 화면에서 장문 설명하지 않는다
- 첫 reward에서 상점까지 미리 설명하지 않는다
- 첫 shop에서 붕괴/상단 합계/오버런까지 한 번에 묶지 않는다

### 프로그래머 handoff

- `seenRewardPrimer`, `seenShopPrimer` 플래그 추가
- 노출은 헤더 문구 재사용이 우선이고, 조건부 class 토글만으로 끝낼 수 있게 구성

### UI handoff

- 기존 헤더 레이아웃을 깨지 않는 선에서 badge + subcopy 추가
- 카드 그리드보다 헤더 문맥을 먼저 읽히게 한다

### 아트 handoff

- reward / shop 목적을 나누는 색 띠(accent strip) 2종이면 충분
- 신규 아이콘이 필요하면 `즉시`, `장기` 의미의 범용 심볼 정도만 준비

---

## 6.4 첫 ending / 첫 overrun primer

### 목표

플레이어가 런 종료 화면에서 **무엇이 남았는지**와 **다음 버튼이 무슨 의미인지**를 헷갈리지 않게 만든다.

### 노출 시점

- 첫 패배 ending 1회
- 첫 승리 ending 1회
- 첫 `canOverrun === true` ending 1회

### 패배 ending 메시지

> 런은 끝나도, 이번에 전진한 페이지만큼 `서고 기록`은 남는다.

### 승리 ending 메시지

> 책을 봉인하면 완주 보너스를 챙기고, 원하면 오버런으로 다음 권을 이어 갈 수 있다.

### 오버런 선택 메시지

> `오버런 계속`은 새 런 시작이 아니라, 지금 만든 흐름을 들고 다음 세그먼트로 넘어가는 선택이다.

### 화면 구성

- ending summary 카드 상단에 1문장 primer 삽입
- `오버런 계속` CTA 근처에는 행동 의미를 다시 한 번 짧게 표기
- 향후 `overrunEvent`가 구현되면 이 primer는 엔딩과 이벤트 페이지 사이 역할이 겹치지 않도록 밀도를 더 낮춘다

### 프로그래머 handoff

- 승리/패배/오버런 가능 여부에 따라 primer 문구 분기
- `overrunEvent` 구현 전후 모두 자연스럽게 읽히게 카피 조건을 분리
- 즉 현재는 `보관함 선택 -> 오버런 계속`, 이후에는 `오버런 계속 -> overrunEvent`로 이어져도 문장이 어색하지 않아야 한다

### UI handoff

- CTA를 가리지 않는 서브카피 배치 우선
- 패배 화면도 `손실`보다 `남는 진척`을 먼저 보게 해야 한다

### 아트 handoff

- victory / defeat / overrun-available 상태를 나누는 소형 배지/라인 장식 정도면 충분
- 오버런 전용 새 메인 일러스트는 범위 밖

---

## 7. 정보 우선순위 규칙

온보딩 카피는 아래 순서를 벗어나면 안 된다.

1. **지금 이 화면에서 무엇을 하면 되는가**
2. **왜 그 행동이 중요한가**
3. **나중에 이해해도 되는 정보는 무엇인가**

예를 들어 첫 전투에서는 `붕괴 시스템의 장기 의미`보다 `이번 턴 주문은 한 번`이 먼저다. 첫 ending에서는 `오버런 설계 철학`보다 `이번 런에서 서고 기록이 남는다`가 먼저다.

즉 좋은 온보딩은 정보를 더 많이 넣는 것이 아니라, **나중 정보는 과감히 뒤로 미루는 것**이다.

## 8. 카피 가이드

### 권장 톤

- 짧고 직접적일 것
- 명령형보다 안내형일 것
- `지금` / `이번 턴` / `이번 런`처럼 문맥을 붙일 것
- 손해 설명도 추상평보다 구체 행동과 연결할 것

### 피해야 할 톤

- 교과서식 장문 설명
- 세계관 과잉 문장
- 초보자를 위축시키는 난이도 경고
- 모든 예외를 한 번에 설명하는 문장

### 좋은 예시

- `이번 턴엔 주문을 한 번만 씁니다.`
- `보상은 다음 전투를 버티기 위한 즉시 선택입니다.`
- `런은 끝나도 서고 기록은 남습니다.`

### 나쁜 예시

- `DiceSpell의 핵심 시스템은 다양한 족보와 상태이상, 붕괴, 상단 합계, 메타 진행이 복합적으로 맞물려...`
- `이 책은 고난도이므로 초보자는 선택하지 않는 것을 권장합니다.`
- `지금부터 오버런 시스템에 대해 설명하겠습니다.`

## 9. 최소 상태 저장 제안

권장 저장 단위는 아래처럼 작게 유지한다.

```json
{
  "uiGuidance": {
    "seenBookSelectPrimer": true,
    "seenBattlePrimer": true,
    "seenFirstSpellHint": true,
    "seenAutoPassHint": true,
    "seenRewardPrimer": true,
    "seenShopPrimer": true,
    "seenDefeatEndingPrimer": true,
    "seenVictoryEndingPrimer": true,
    "seenOverrunPrimer": true
  }
}
```

원칙:

- 런 세이브가 아니라 메타/환경 설정 쪽에 두는 편이 자연스럽다
- `첫 런 한 번만`과 `도움말 다시 보기`는 분리 가능하게 설계한다
- 저장 누락 시 치명적 오류가 아니라 **힌트가 한 번 더 보이는 수준**이어야 한다

## 10. owner별 실제 handoff

### 10.1 프로그래머가 넘겨야 하는 것

1. primer 노출용 최소 상태 플래그
2. 각 surface별 조건부 render hook
3. `overrunEvent` 도입 전후에도 모순 없이 읽히는 ending primer 분기
4. primer 비노출 시 기존 플로우가 100% 유지된다는 보장
5. smoke 또는 최소 렌더 검증 포인트

### 10.2 UI 오너가 넘겨야 하는 것

1. 도서관 / 전투 / reward / shop / ending primer 레이아웃
2. 기존 핵심 인터랙션을 가리지 않는 배치 규칙
3. compact / expanded 두 밀도 중 하나를 surface별로 명시한 시안
4. `다시 보지 않기` 또는 `알겠음` 인터랙션 처리안

### 10.3 아트 오너가 넘겨야 하는 것

1. primer 공통 backplate 또는 ribbon 1세트
2. `추천`, `주문 1회`, `즉시`, `장기`, `남는 진척`, `오버런` 정도의 소형 아이콘/배지 세트
3. 기존 색 체계와 충돌하지 않는 accent 규칙

### 10.4 QA가 넘겨야 하는 것

1. 첫 런 기준 primer가 의도한 surface에서만 뜨는지 확인
2. 한 번 본 후 반복 노출이 멈추는지 확인
3. 저장 누락/호환 이슈가 나도 진행이 막히지 않는지 확인
4. `overrunEvent` 전후 버전 모두에서 ending primer 문맥이 깨지지 않는지 확인

## 11. 구현 순서 제안

이번 온보딩은 아래 순서가 가장 안전하다.

### Step 0. 카피/위치만 먼저 고정

- surface별 한 줄 목적 문구 확정
- 기존 헤더/패널 어디에 얹을지 결정
- 새 구조가 아니라 재사용 구조를 우선 확인

### Step 1. 도서관 + 첫 전투 primer

- 가장 큰 초반 이탈 지점을 먼저 줄인다
- `책 카드 읽는 순서`와 `전투 1턴 구조`를 먼저 닫는다

### Step 2. reward / shop primer

- `즉시` vs `장기` 문맥 차이를 붙인다

### Step 3. ending / overrun primer

- `무엇이 남았는지`, `버튼이 무엇을 뜻하는지`를 닫는다
- 이 단계는 `overrunEvent` 구현 일정과 카피 충돌이 없는지 함께 본다

### Step 4. 반복 노출/저장/도움말 재오픈 정리

- 기본 플로우를 먼저 닫은 뒤 UX polish로 정리

## 12. smoke / QA 기준

### 최소 자동 검증

- 첫 런 상태에서 primer 관련 render 분기가 오류 없이 돈다
- primer 비활성 상태에서도 기존 화면이 그대로 열린다
- `overrunEvent` 구현 전후에 ending primer 카피 분기가 깨지지 않는다

### 최소 수동 검증

1. 새 프로필/초기 메타에서 책 선택 primer 1회 노출
2. 첫 전투 진입 primer 노출 후 첫 주문 힌트 노출
3. 주문 불가 자동 종료가 발생하면 관련 힌트 노출
4. 첫 reward / 첫 shop에서 목적 문구 노출
5. 첫 패배 ending에서 `서고 기록이 남는다`가 읽힘
6. 첫 승리/오버런 가능 ending에서 `이어 간다` 문맥이 읽힘
7. 한 번 본 뒤 같은 primer가 계속 반복되지 않음
8. 저장을 지우거나 플래그가 누락돼도 진행은 막히지 않음

## 13. 완료 정의(DoD)

이 온보딩 pass는 아래가 충족되면 완료로 본다.

- 첫 런 10분 안에 필요한 4개 surface primer가 bounded하게 붙었다
- primer가 기존 핵심 인터랙션을 가리지 않는다
- 책 선택 / 전투 / reward·shop / ending 각각에서 `지금 뭘 보면 되는가`가 1문장으로 읽힌다
- `overrunEvent` 구현 전후 카피 충돌이 없다
- 저장/반복 노출 이슈가 진행을 막지 않는다

반대로 아래는 아직 완료가 아니다.

- 모든 시스템 설명을 다 넣은 것
- 별도 튜토리얼 전투나 연습 모드를 만든 것
- 접근성/로컬라이징 전체가 정리된 것

즉 DoD는 `거대한 튜토리얼 완성`이 아니라, **첫 런 초반 해석 비용을 눈에 띄게 낮추는 최소 가이드 구조가 붙었는가**다.

## 14. 리스크 & 후속 과제

### 리스크 1. 설명이 많아져 템포를 다시 해칠 위험

- 영향: 첫 런 이탈은 줄어도 DiceSpell 특유의 압축된 템포가 무뎌질 수 있다.
- 대응: 각 surface는 한 화면 한 메시지 원칙을 지키고, modal보다 inline/card를 우선한다.

### 리스크 2. `overrunEvent` 구현과 카피 문맥이 충돌할 위험

- 영향: 엔딩에서 이미 `다음 권으로 넘어간다`를 너무 많이 말하면 새 이벤트 페이지의 의미가 약해질 수 있다.
- 대응: ending primer는 `무엇이 남는가`와 `버튼 의미`에만 집중하고, 장면 연출은 `overrunEvent` 쪽에 남긴다.

### 리스크 3. 책 추천이 메타 서열처럼 읽힐 위험

- 영향: 초보자용 가이드가 특정 책 강요처럼 읽히면 전체 다양성이 줄어든다.
- 대응: `쉬움/어려움`보다 `읽기 쉬운 운영`, `주의할 실수` 중심으로 표현한다.

### 리스크 4. 저장 플래그가 늘어 운영 복잡도가 커질 위험

- 영향: 사소한 UX 때문에 저장 호환 리스크가 불필요하게 커질 수 있다.
- 대응: primer 플래그는 최소 개수로 유지하고, 누락 시 재노출만 일어나게 설계한다.

### 다음 후속 과제

1. `overrunEvent` 실제 구현 일정과 겹치지 않게 ending primer 카피를 같은 owner가 함께 검토한다.
2. 온보딩 pass를 실제로 붙일 때는 Step 1(`책 선택 + 첫 전투`)만 먼저 구현해도 된다.
3. 이후 FQA 관찰에서는 `첫 전투 첫 3턴 안에 무엇을 해야 하는지 감이 오는가`, `책 선택에서 과잉 정보로 멈추지 않는가`를 별도 체크한다.

## 15. 최종 결론

DiceSpell에 지금 필요한 온보딩은 `게임을 처음부터 다시 설명하는 거대한 튜토리얼`이 아니다.

필요한 것은 아래 네 문장만 제때 보여 주는 것이다.

- 이 책은 어떻게 읽는가
- 이번 턴엔 무엇을 하는가
- 이 선택은 당장 도움인가, 장기 투자인가
- 이번 런에서 무엇이 남았는가

이 네 문장을 올바른 순간에 보여 줄 수 있으면, 지금까지 쌓아 온 전투/책/보상/오버런의 장점은 훨씬 덜 새고 더 빨리 읽히기 시작한다.
