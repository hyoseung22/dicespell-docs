1. Overview

한눈에 보는 강화석 동작

강화석은 현재 빌드에서 족보의 기본 피해를 영구적으로 누적 강화하는 보상 축이다. 구현 문법은 단순하다. 각 족보 슬롯은 powerLevel을 가지며, 이 값이 1 오를 때마다 해당 족보가 매치되었을 때만 기본 피해 +4를 추가로 얻는다.

현재 빌드 정의

강화석은 무작위 족보 1~5개를 골라 powerLevel을 올리는 즉시형 보상이다. 직접 강화석 아이템 자체는 항상 powerGain = 1을 사용한다.

기획적으로 읽는 의미

정령석이 속성 방향을 열어 주는 축이라면, 강화석은 같은 족보를 더 무겁게 마감시키는 화력 축이다. 룰을 바꾸지 않고 결과를 증폭한다.

선택 UX 의미

강화석은 즉시 적용형이며 별도 타깃 선택 단계가 없다. 대신 reward/shop 화면 안에서 무엇이 바뀌었는지를 피드백 카드로 바로 보여 준다.

전투 시스템 의미

강화석은 MP, 타깃 타입, 속성, 잠금 규칙을 건드리지 않는다. 오직 매치 성공 시 기본 피해에만 더해진다.

질문형 핵심 정리

질문현재 빌드 답변
무엇을 강화하는가족보 슬롯별 powerLevel. 결과적으로 매치 성공 시 기본 피해가 증가한다.
얼마나 오르는가레벨 1당 +4. 직접 강화석 1회는 기본적으로 +1레벨.
누가 타깃을 고르는가플레이어 직접 선택 없음. 전체 족보에서 무작위 고유 선택.
정령석처럼 빈 슬롯 우선인가아니다. 강화석은 모든 족보를 동일한 랜덤 풀로 취급한다.
매치 실패 족보도 이득을 받는가아니다. 해당 족보가 실제로 매치되었을 때만 피해 보너스가 더해진다.
상점 가격은 어떻게 계산되는가55 + (타깃 수 - 1) × 18. 타깃 수가 많을수록 가격이 오른다.
직접 강화석 외에도 같은 로직을 쓰는가그렇다. 전리품 번들, 몬스터 트로피, 일부 책 완주 보너스도 같은 upgradeHands 경로를 재사용한다.
시각 피드백은 있는가있다. reward/shop 직후 hand-change feedback 카드, 로그, 전투 카드의 강화 +n 표기가 동기화된다.

2. State Model

데이터 모델

강화석은 별도 장비나 소모품 인벤토리로 저장되지 않는다. 현재 빌드는 각 족보 슬롯에 직접 powerLevel을 저장하고, 전투 계산 시 이 수치를 읽어 피해 보너스로 변환한다.

필드 / 상수현재 빌드 의미비고
HAND_UPGRADE_DAMAGE레벨 1당 추가 기본 피해량현재 값 4
handSlots[handId].powerLevel해당 족보의 누적 강화 레벨기본값 0
handPowerDamageBonus(slot)max(0, powerLevel) × 4음수 불가
applyUpgradeStone(...)실제 강화 적용 공통 함수보상/상점/번들/완주 보너스 공통 사용

초기값과 저장 규칙

각 족보 초기 상태 { locked: false, element: "None", powerLevel: 0 } 강화 후 저장 규칙 powerLevel = clamp(currentPowerLevel + powerGain, 0, 99)
설계 포인트: 강화석은 “족보 자체를 변환”하는 시스템이 아니라 “족보 슬롯에 누적 스택을 새기는 시스템”이다. 따라서 저장/세이브/요약/전투 미리보기 모두 같은 상태값을 참조한다.

3. Spawn Rules

생성 규칙

강화석은 현재 빌드에서 기본 전투 보상, 상점 오퍼, 번들 보상, 책 완주 보너스라는 네 경로로 들어온다. 직접 강화석 오퍼의 생성 확률과 가격은 고정식이다.

4. Application Flow

적용 절차

직접 강화석이든 upgradeHands 번들이든, 최종 적용은 전부 같은 공통 함수로 들어간다. 따라서 “어디서 받았는가”보다 “어떤 targetCount와 powerGain이 들어왔는가”가 핵심이다.

보상 선택 / 상점 구매 / 번들 효과 → applyUpgradeStone() → 무작위 고유 족보 선택 → powerLevel += powerGain → 로그 / UI 요약 갱신

실제 적용 알고리즘

targetCount = clamp(payload.targetCount, 1, HAND_LAYOUT.length) powerGain = clamp(payload.powerGain, 1, 99) selected = shuffle(HAND_LAYOUT).slice(0, targetCount) for each handId in selected: handSlots[handId].powerLevel = clamp(current + powerGain, 0, 99)

중요한 현재 빌드 해석

항목현재 빌드 동작정령석과의 차이
선택 풀항상 전체 HAND_LAYOUT 대상정령석은 빈 속성 슬롯 우선 로직이 있다.
중복 여부한 번의 적용 안에서는 중복 없음두 시스템 모두 한 번의 적용에서는 고유 선택이다.
플레이어 직접 지정없음없음
직접 효과량기본 powerGain = 1정령석은 피해량 대신 속성을 바꾼다.
즉시 반영즉시 상태에 기록속성 변경과 동일하게 즉시 저장된다.
주의: 강화석은 현재 빌드에서 “약한 족보를 우선 보정”하거나 “비강화 족보를 우선 고르는” 보정이 없다. 이미 강화된 족보가 다시 뽑히는 것도 허용한다.

5. Combat Effect

전투 수치 반영

강화석은 피해 공식을 크게 갈아엎지 않는다. 기존 족보 판정으로 얻은 baseDamage에만 추가량을 더한다. 따라서 동일 족보를 같은 주사위 구조로 맞췄을 때 더 무거운 피니시가 나오는 방식이다.

실제 적용 공식

matchInfo.matched ? baseDamage = matchInfo.baseDamage + (powerLevel × 4) : baseDamage = 0 finalDamage = floor(baseDamage × totalMultiplier × affinity × statusMultiplier × passiveMultiplier × shieldedMultiplier)

중요한 현재 빌드 의미

항목현재 빌드 의미
매치 성공 여부매치 실패면 강화 수치가 있어도 피해는 0이다.
적용 시점기본 피해 단계에서 먼저 더해지고, 이후 속성 상성/아티팩트/붕괴 배율과 함께 최종 피해로 확장된다.
영향 없는 항목MP 비용, 족보 잠금, 타깃 타입, 속성, 상단 합계, 리롤 구조는 바뀌지 않는다.
전투 미리보기세션 요약 UI도 이미 강화 보너스를 포함한 baseDamage를 보여 준다.

예시

상황현재 빌드 해석
기본 피해 18인 족보, 강화 0매치 성공 시 기본 피해 18
기본 피해 18인 족보, 강화 +1매치 성공 시 기본 피해 22
기본 피해 18인 족보, 강화 +2매치 성공 시 기본 피해 26
강화 +2지만 매치 실패피해 0. 강화 수치가 있어도 발동하지 않음

8. Edge Cases

예외 / 주의사항

항목현재 빌드 해석
비강화 우선 선택 없음강화되지 않은 족보를 우선 보정하지 않는다. 이미 강화된 족보가 또 뽑힐 수 있다.
잠긴 족보도 대상 가능선택 풀은 HAND_LAYOUT 전체다. 잠금 여부는 적용 풀에서 제외 조건이 아니다.
속성 여부 무관속성이 이미 새겨진 족보도 강화 대상이 될 수 있다.
한 번의 적용 내 중복 없음shuffle(...).slice(...)라서 동일 적용 안에서는 중복이 없다.
여러 번 적용 간 중복 가능다음 강화석/번들에서는 같은 족보가 다시 뽑힐 수 있다.
직접 강화석 powerGain항상 1. 다만 내부 번들/완주 보너스는 2 이상을 넣을 수 있다.
피드백 카드 범위reward/shop 액션에서 가장 명확하게 노출된다. 내부 효과 재사용 경로는 상태는 같아도 같은 카드가 항상 보장되지는 않는다.
상한powerLevel은 99에서 클램프된다.

직접 강화석 적용선택 즉시 로그와 hand-change feedback 카드가 뜨고, 최소 1개 이상의 족보 powerLevel이 오른다.
