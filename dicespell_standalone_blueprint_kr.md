# DiceSpell 스탠드얼론 제작 청사진

## 문서 성격

이 문서는 `DiceSpell_GDD v2.0`과 현재 스탠드얼론 구현을 기반으로, 실제 제작 우선순위와 규칙 확정을 정리한 `실행 리비전`이다.

- 목적: 문서가 많아도 팀의 판단이 흔들리지 않도록, 지금 무엇을 고정하고 무엇을 나중으로 미룰지 명확히 한다.
- 기준: `현재 구현 가능한 범위`와 `플레이 감각을 가장 크게 바꾸는 항목`을 우선한다.
- 사용 방식: 원안 GDD를 대체하지 않고, 실제 개발 결정의 기준 문서로 사용한다.

## 문서 선행 원칙

- DiceSpell의 모든 작업은 `개발 추적 기획서`와 `종합 게임 기획서`를 먼저 갱신한 뒤 진행한다.
- 구현 전 선행 문서:
  - `docs/dicespell_development_tracker_kr.html`
  - `docs/DiceSpell_GDD_sync.html`
- `docs/DiceSpell_GDD_sync.html`을 바꿨다면 `/Users/jeonghyoseung/Downloads/DiceSpell_GDD.html`도 같은 턴에 동기화한다.
- 공개 문서에 영향이 있다면 `docs/gdd.html`과 `docs/development-tracker.html`도 같은 턴에 재동기화하고 `scripts/publish_docs_portal.sh`로 `https://hyoseung22.github.io/dicespell-docs/`를 재배포한다.
- 반복 구현 writer는 `OpenClaw DiceSpell execution loop 30m` 하나만 사용하고, 다른 반복 자동화는 read-only 또는 review-only 역할로 분리한다.
- 반복 구현 자동화는 시작 전에 저장소 루트에서 `./scripts/dicespell_run_lock.sh acquire <owner>`로 writer lock을 먼저 획득하고, 종료 전에 반드시 해제한다.
- Codex는 DiceSpell 저장소의 버전 관리자 역할을 겸한다. 따라서 새 요청은 구현 전에 자동화 충돌, 문서 일관성, 검증 재현성, 커밋 단위, 배포 안전성에 미치는 영향을 먼저 평가하고, 위험이 있으면 더 안전한 대안으로 재구성한다.
- 범위가 바뀌면 코드보다 먼저 선행 문서를 다시 수정한다.

## 제품 관점

- 목표: 야추 기반 주문 전투의 정체성을 유지하면서, 데스크톱 프로토타입을 `반복 플레이 가능한 전략 로그라이크`로 끌어올린다.
- 포지셔닝: `주사위 족보 전투`, `붕괴 리스크 운영`, `책 단위 런 메타`가 결합된 싱글플레이 전략 RPG.
- 플레이 감정선:
  - 도서관에서 어떤 책을 고를지 고른다.
  - 전투에서 한 턴의 결정을 압축한다.
  - 붕괴와 잠금이 쌓이며 압박이 오른다.
  - 보상과 상점으로 빌드 방향을 굳힌다.

## 확정된 핵심 원칙

### 1. 턴 구조는 `한 턴 = 한 주문`으로 고정한다

- 플레이어 턴은 `Roll -> Lock -> Spell 1회` 또는 `Pass`로 끝난다.
- 주문이 발동되면 바로 결과 처리, 붕괴 판정, 상단 합계 체크를 거쳐 몬스터 턴으로 넘어간다.
- 첫 롤 이후 현재 MP로 사용할 수 있는 잠기지 않은 족보가 하나도 없으면, 턴은 자동으로 몬스터 턴으로 넘어간다.
- 이 결정은 전투를 `연속 행동 콤보형`이 아니라 `한 번의 압축된 선택` 중심 게임으로 정의한다.

### 2. 족보 잠금은 `페이지 단위 압박 자원`으로 유지한다

- 족보 잠금은 주문 사용 후 발생한다.
- 잠금은 현재 페이지 동안 유지된다.
- 페이지 전환 시 전부 해제된다.
- 따라서 긴장감은 `전투 내 소모 압박`에 두고, 런 전체를 묶는 영구 페널티로는 사용하지 않는다.

### 3. 상단 합계 보너스는 `기준값 이상`에서 즉시 개방한다

- 상단 1~6의 총합이 기준값에 `도달하거나 초과`하면 보너스 드래프트가 열린다.
- 이 드래프트는 전투마다 1회만 열리며, 현재 기본 기준값은 `55`다.
- 싱글 족보 점수는 누적 가산이 아니라 `최신 기록`으로 덮어쓴다. 족보 일치면 해당 상단 점수를 기록하고, 불일치로 시전하면 그 칸은 `0`으로 갱신된다.
- 주문 해석 뒤 패배나 전투 종료가 먼저 확정되면 드래프트는 뜨지 않는다. 전투가 계속될 때만 10종 라이브러리 중 랜덤 3택 1이 열린다.
- 보너스를 고르면 효과를 즉시 적용하고, 처치 재판정까지 끝낸 뒤 전투가 남아 있으면 붕괴 판정과 몬스터 턴으로 넘어간다.
- 기준은 기본 `55`이며, 로비 밸런스 랩에서 조정 가능하다.
- 상단 합계는 싱글 족보를 단순 초반 카드가 아니라 빌드 목표로 만드는 장치다.

### 4. 보상과 상점의 역할을 분리한다

- 전투 보상:
  - 다음 1~2전투의 생존과 전술 보정을 담당한다.
  - 정령석, 강화석, 회복처럼 즉시 적용되는 선택을 준다.
  - 즉시 적용되는 선택은 결과가 화면 안에서 바로 읽혀야 한다. 정령석/강화석은 어떤 족보가 바뀌었는지, 무엇이 새로 기대되는지 같은 슬롯에서 피드백한다.
- 상점:
  - 런의 장기 빌드 방향을 담당한다.
  - 유물 중심의 중장기 투자 공간으로 유지한다.
  - 현재빌드의 오퍼 구조는 `유물 3 / 정령석 3 / 강화석 3`을 기준으로 맞추고, 각 묶음이 같은 화면에서 비교 가능해야 한다.
- 이후 업데이트도 이 역할 분리를 해치지 않는 방향으로 설계한다.

### 5. 마도서는 `수치 차이`보다 `운영 차이`를 만들어야 한다

- 각 마도서는 최소 3가지로 정체성이 읽혀야 한다.
  - 시작 자원/스타터 유물
  - 추천 운영 축
  - 명시적인 플레이 규칙 또는 보상 구조
- 로비에서 실제 선택은 전체 해금 목록을 그대로 나열하지 않고, 해금된 책 중 일부를 랜덤 제안 풀로 보여 주는 방식이 더 적합하다.
- 대신 전체 책, 유물, 몬스터 정보는 선택 보드와 분리된 `보관함` 백과사전에서 항상 다시 볼 수 있어야 한다.
- 현재의 `스타팅 프리셋` 느낌을 줄이고, 책 선택이 런의 전략 선언처럼 보이게 만든다.

### 6. 런 종료는 손실이 아니라 메타 진척으로 귀결되어야 한다

- 실패하거나 도중에 런을 접어도 완료한 페이지 수만큼 `서고 기록`을 보상으로 준다.
- 책을 끝까지 클리어하면 즉시 `오버런`으로 이어서 다음 책 구간을 붙여 플레이할 수 있게 한다.
- 누적된 서고 기록은 다음 두 축을 해금하는 데 쓴다.
- 새로운 마도서
- 새 런에 들고 갈 수 있는 유산 아티팩트
- 즉, 개별 런은 끝나도 플레이어의 장기 진척은 항상 남아야 한다.

### 7. 몬스터 배치는 `난이도 기반 랜덤 스폰풀`로 운영한다

- 전투 페이지는 `segmentPage`와 `overrunLevel`을 기준으로 현재 전투 난이도를 계산한다.
- 노멀 몬스터, 보스, 최종 보스는 각각 별도의 스폰풀을 가진다.
- 각 페이지는 자신의 난이도에 가까운 후보군에서 가중 랜덤으로 적을 뽑는다.
- 같은 책 테마 몬스터는 더 자주 나오게 가중치를 주되, 다른 테마 몬스터도 섞여 반복감을 줄인다.
- 2~3마리 전투는 `전열 / 술사 / 교란` 편성 역할 슬롯을 우선 채우도록 가중치를 준다.
- 이미 전장에 있는 역할은 가중치를 낮춰, 같은 성격의 적만 겹쳐 나오는 상황을 줄인다.
- 즉, 책 분위기는 유지하되 `같은 책이면 같은 적만 반복`되는 구조는 피한다.

## MVP 우선순위

### 1순위. 콘텐츠 완성도와 런 연결감

- 이제 코어 규칙은 기본적으로 확정된 상태로 본다. 다음 단계는 `무엇을 더 할 수 있는가`보다 `왜 다음 페이지와 다음 런을 계속 하고 싶은가`를 더 선명하게 만드는 것이다.
- 책 클리어 보상, 오버런, 후속 페이지, 적/보상/메타가 서로 이어지는 흐름을 우선 만든다.
- 새 콘텐츠는 단독 추가보다 `이전 선택 -> 현재 보상 -> 다음 기대`가 살아나는 구조를 먼저 본다.

### 2순위. 재미/FQA와 시스템 유기성

- 각 30분 슬롯은 가능한 한 `재미가 실제로 늘어나는 콘텐츠 1개`를 만드는 쪽으로 사용한다.
- FQA 기준은 `기억에 남는 순간이 생기는가`, `선택한 책/보상이 실제 플레이 성격을 바꾸는가`, `새 콘텐츠가 기존 시스템과 자연스럽게 연결되는가`다.
- 이 단계에서는 수치 최적화보다 플레이 감정선, 런 템포, 보상 기대감, 적/보상 상호작용을 더 중요하게 본다.

### 작업 선택 체크리스트

- 반복 작업은 아래 질문에 먼저 답하고 시작한다.
  1. 이 변경이 `이전 선택 -> 현재 보상 -> 다음 기대`를 더 강하게 만드는가
  2. 플레이 감정선, 기억점, 놀람 포인트, 다시 하기 욕구 중 적어도 하나를 늘리는가
  3. 필요한 UX 피드백을 같은 슬롯에서 최소 보강할 수 있는가
  4. 밸런스 검증은 `최소 허들 확인` 수준으로 충분한가
  5. 문서/검증/배포/커밋까지 한 사이클로 끊을 수 있는가
- 위 질문에서 `1~2`가 약한 작업은 blocker가 아닌 이상 후순위다.
- 즉, 지금 단계의 좋은 작업은 `숫자를 예쁘게 만드는 일`보다 `한 런을 더 하게 만드는 작은 콘텐츠 축 1개를 끝내는 일`이다.

### 3순위. UI/UX 후속 정리

- UX는 후순위지만 미루기만 하지 않는다. 새 콘텐츠와 직접 맞닿는 안내, 로그, 표식, 버튼 카피는 같은 사이클에서 최소 보강한다.
- 플레이어가 `방금 재미있었던 이유`와 `왜 손해를 봤는지`를 바로 읽을 수 있게 한다.
- 튜토리얼, 툴팁, 접근성은 콘텐츠 축이 더 선명해진 뒤 묶어서 정리한다.

### 4순위. 밸런스 최소 허들과 회귀 방지

- 밸런스는 당분간 `최적화 대상`이 아니라 `최소 플레이 가능 허들`로 취급한다.
- 최소 허들은 다음 네 가지다:
  - smoke와 기본 검증이 통과할 것
  - 명백한 soft-lock, 무대응 패턴, 즉시 포기 유발 스파이크가 없을 것
  - 새 콘텐츠의 핵심 보상이 실제 플레이에서 체감될 것
  - 기존 책/런이 이번 변경으로 눈에 띄게 망가지지 않을 것
- 비교 시뮬레이션은 필요할 때만 쓰되, 새 콘텐츠를 가로막는 절대 우선순위로 사용하지 않는다.

### 5순위. 장기 메타와 확장 보상

- 현재 구현된 `서고 기록 -> 책/유산 해금` 구조를 더 풍부하게 만들되, 영구 수치 누적보다 `다음 런의 선택지`를 넓히는 보상에 집중한다.
- 현재는 모든 책에 `완주 보너스`를 붙여, 책을 끝낸 뒤 오버런을 이어갈 즉시 보상을 제공한다.
- 장기 캠페인 40페이지 복원, 다중 세이브 슬롯, 영구 능력치 메타는 위 우선순위가 안정된 뒤 다시 본다.

## UI/UX 원칙

- 매 턴마다 `지금 무엇을 할 수 있는지`를 버튼과 상태 텍스트로 선명하게 보여준다.
- `잠금`, `MP 부족`, `족보 불일치`, `이번 턴 주문 1회`를 숨기지 않는다.
- 전투에서 실수했을 때 왜 손해였는지 로그와 패널에서 바로 읽히게 한다.
- 상단 합계와 붕괴는 서브 시스템이 아니라 주요 의사결정 패널로 취급한다.

## 개발 관점

- 기술 스택:
  - Electron 기반 데스크톱 앱
  - 렌더러는 순수 HTML/CSS/JS 모듈
  - 저장은 localStorage 기반
- 코드 구조:
  - `dice-spell-standalone/src/data.js`: 마도서, 몬스터, 족보, 유물 데이터
  - `dice-spell-standalone/src/game-engine.js`: 순수 상태 전이와 계산
  - `dice-spell-standalone/src/app.js`: 렌더링과 상호작용 바인딩
- 데이터 구조 목표:
  - `grimoires`, `stages`, `monsters`, `monster actions`, `hands`, `artifacts`, `collapse tiers`를 외부 데이터로 정규화할 수 있게 유지한다.

## GDD 반영 지침

원안 GDD를 다음 기준으로 갱신한다.

1. `3.1 전투 흐름`
   - 턴당 주문 1회, 주문 불가 시 자동 턴 종료를 명시한다.
2. `3.5 상단 합계 보너스`
   - 기준값 `이상`에서 발동하는 것으로 통일한다.
3. `8. 상점 & 보상`
   - 전투 보상은 단기 전술, 상점은 장기 빌드라는 역할 차이를 명시한다.
4. `9. 스테이지 구조`
   - 현재는 15/30페이지 구조를 유지하고, 40페이지 모드는 후속 확장으로 남긴다.
5. `15. 리스크 & 후속 과제`
   - 시스템 리스크와 콘텐츠 확장을 `현재 대응`, `후속 대응`으로 나눠 관리한다.

## 제작 로드맵

### 단계 1. 코어 전투 정리

### 단계 2. 마도서 정체성 강화

153. 핵심은 이제 Run Table oath 축의 남은 공백이 `첫 10분에 무엇부터 시작하지?`를 넘어 `좋아, 그 archive와 kickoff 순서는 알겠어. 그런데 실제 코드를 열기 직전에 정확히 어느 locator를 다시 찾고 어디서 멈춰야 ending helper / reward-shop side family / queue closing wording까지 같이 열리지 않지?`라는 점을 oath-only semantic anchor 언어로 잠그는 것이다. 즉 다음 실제 oath refinement는 반드시 `source-of-truth map -> gap ledger -> queue readiness board -> oath live kickoff -> oath semantic anchor packet -> manifest fill -> oath required artifacts -> oath candidate review -> queue closing packet` 순서로 읽고, `root / role / cta / boundary / non-change` note와 `overview / banner-focus` 질문을 먼저 적은 뒤에만 oath live edit를 시작해야 한다. 이어 이제 마지막 운영선에서는 `queue readiness board -> queue closing packet -> queue signoff ladder -> follow-up reopen packet` 순서까지 같은 source-of-truth로 읽고, 프로그래머/UI/아트는 stage evidence 제출과 `follow-up needed` 메모까지만, QA는 `queue-closing review-ready`와 `follow-up reopen proposal`까지만, PM은 `Run Table token queue closing`과 `Run Table token follow-up reopen`까지만 남겨 ownership drift와 rollback처럼 보이는 reopen drift를 함께 막아야 한다.
154. 새 리스크는 kickoff, required artifacts, candidate review, queue closing packet까지 다 정리된 상태에서 오히려 implementer와 reviewer가 locator를 다시 감으로 찾다가 `banner를 보다가 result table 전체도`, `carry를 보다가 reward/shop side family도`, `continue-overrun CTA를 보다가 queue closing wording도` 같이 열어 마지막 stage boundary를 흐리는 것이다. 따라서 다음 실제 oath turn은 반드시 새 semantic anchor packet의 `surface root / role / CTA / boundary / capture proof` 5축을 같은 stage note로 재사용하고, overview / banner-focus evidence가 oath-only 질문과 연결되기 전에는 candidate/green/queue closing을 쓰지 않아야 한다. 새 리스크는 semantic locator까지 잠근 뒤에도 archive file routing이 없으면 implementer/UI/아트/QA/PM이 `templates/` / `stubs/` / `examples/manifest_example_oath_banner.md`를 실제 제출 경로처럼 다루거나, live root 없이 candidate/green 문장을 먼저 쓰는 것이다. 따라서 다음 실제 oath turn은 반드시 새 archive file map packet의 `source docs -> starter files -> live archive destination -> owner fill order` 4층을 먼저 잠그고, `docs/artifacts/run-table-token-delivery/oath-banner/` 경로 밖에는 live evidence를 남기지 않아야 한다.
