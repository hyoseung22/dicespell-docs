# DiceSpell 오버런 전리품 이벤트 페이지 handoff spec

## 빠른 안내

### 3줄 요약
- 이 문서는 `오버런 전리품 보관함` 다음에 붙는 `overrunEvent`를 **무엇을 위한 phase인지**부터 고정하는 원본 handoff다.
- 프로그래머는 phase 경계와 상태 책임을 먼저 보고, UI·아트는 `선택 화면`이 아니라 `선택을 들고 넘어가는 장면`이라는 점을 먼저 보면 된다.
- 세부 구현은 뒤 문서로 내려가더라도, 이 문서는 끝까지 **범위와 의미를 흔들리지 않게 하는 기준 문서**로 읽는다.

**한 줄 목표:** 오버런 전리품 선택을 `다음 권으로 넘어가기 직전의 짧은 이벤트 페이지 경험`으로 승격한다.

### 왜 이 문서를 먼저 보나
- 이 문서는 새 메카닉 추가가 아니라 장면 경계 추가라는 점을 첫 화면에서 분명히 만든다.
- 노션/컨플에 붙여 넣어도 `ending 확장인가, 새 phase인가`를 바로 판정할 수 있게 하기 위한 문서다.

### 누가 어디부터 읽으면 되나
| 역할 | 먼저 볼 부분 | 바로 이어서 볼 문서 |
| --- | --- | --- |
| 프로그래머 | phase 정의, 진입/미진입 조건 | `implementation packet`, `integration workorder` |
| UI | 장면 목적, 카드 1장 구조, CTA 의미 | `screen packet`, `content deck` |
| 아트 | bounded 무드와 자산 최소선 | `screen packet`, `production cutsheet` |
| QA/PM | 포함/제외 범위, skip 포함 장면 처리 | `acceptance matrix`, `summary` |

## 문서 목적

이 문서는 현재 구현된 `오버런 전리품 보관함`을 **실제 페이지 구조**로 한 단계 더 밀어 붙일 때, 프로그래머 / UI 오너 / 아트 오너가 서로 다른 그림을 보지 않도록 기준을 고정하는 handoff 문서다.

지금까지는 아래 연결이 이미 구현됐다.

- late normal / overrun normal 보상에 `책 테마 전리품` 추가
- `기록 골렘` / `붕락 정령` 처치 시 적 전리품 추가
- 책 클리어 엔딩에 `오버런 전리품 보관함` 3택 1 추가

하지만 아직 부족한 점이 있다.

- 플레이어가 전리품을 고른 뒤 **실제로 다음 권으로 넘어가는 장면**이 페이지 단위로 존재하지 않는다.
- 구현자가 이 다음 단계를 잡으려 하면 `엔딩 화면을 조금 더 꾸미는 것인지`, `새 phase를 만드는 것인지`, `맵 위 이벤트 노드인지`가 모호하다.
- UI/아트도 어떤 화면을 준비해야 하는지, `카드 1장 강조형`인지 `선택 2갈래 이벤트`인지 기준이 없다.

이 문서의 결론은 간단하다.

> 다음 bounded 구현은 `새 전리품 메카닉`을 더 얹는 것이 아니라, 이미 선택한 전리품을 **오버런 시작 직전의 짧은 이벤트 페이지**로 시각화하는 것이다.

즉 이번 handoff는 **현재 메카닉을 더 크게 바꾸지 않으면서도 기억점을 페이지 구조로 승격**시키는 데 목적이 있다.

## 1. 문제 정의

현재 빌드의 강점은 이미 분명하다.

- `새 위협 -> 새 보상 기대` 축이 late normal부터 작동한다.
- 특수 적 전리품과 책 테마 전리품이 오버런 진입 직전까지 이어진다.
- 엔딩의 `오버런 전리품 보관함`은 `이번 세그먼트에서 모은 잔해를 다음 권으로 들고 간다`는 감각을 만든다.

현재 가장 큰 빈칸은 다음이다.

1. **엔딩과 다음 세그먼트 첫 전투 사이에 장면적 완충이 없다.**  
   선택은 했지만, 플레이어는 그 선택을 `들고 넘어간다`는 장면을 아직 페이지로 경험하지 않는다.
2. **역할별 handoff 기준이 없다.**  
   프로그래머는 phase/state 경계가, UI는 화면 정보 구조가, 아트는 필요한 카드/백플레이트/아이콘 범위가 모호하다.
3. **bounded 구현 범위가 흐려질 위험이 있다.**  
   잘못 들어가면 새 이벤트 메카닉, 추가 선택지, 추가 보상 테이블, 새 삽화 파이프라인까지 한 번에 커진다.

따라서 다음 구현은 `더 큰 시스템`이 아니라 **짧고 확실한 세그먼트 경계 이벤트**여야 한다.

## 2. 목표 경험

플레이어 경험 목표는 아래 한 문장으로 정리한다.

> 책을 막 봉인한 직후, 방금 건진 전리품 하나를 책상 위에 올려두고 다음 권 첫 장을 펼치는 짧은 의식을 치른다.

이 경험이 성립하려면 아래가 필요하다.

- 플레이어는 `내가 방금 고른 전리품`이 무엇인지 다시 본다.
- 그 전리품이 `왜 다음 권으로 이어지는지`를 짧은 문장으로 읽는다.
- `계속` 버튼을 누르는 순간, 선택이 실제로 적용됐다는 확신을 받는다.
- 이 장면은 길지 않아야 한다. 다음 전투 템포를 끊으면 안 된다.

즉 이 페이지는 **새 선택지의 양**이 아니라 **선택을 기억점으로 고정하는 연출 페이지**다.

## 3. 이번 spec의 범위

### 포함

- 오버런 진입 직전에 1회만 나타나는 별도 이벤트 페이지 정의
- 새 runtime phase 정의
- 해당 페이지의 정보 구조, 버튼 구조, 로그/카피 구조 정의
- 전리품 출처별(책 테마 / 특수 적 전리품) 시각 강조 규칙 정의
- 프로그래머 / UI / 아트 handoff 정의
- smoke 기준 정의

### 제외

- 새 전리품 타입 추가
- 전리품 선택을 다시 굴리거나 강화하는 추가 메카닉
- 맵 위 별도 노드 그래프 확장
- 오버런 전용 신규 삽화 제작을 전제로 한 대형 아트 작업
- 밸런스 재조정용 시뮬레이션 확대

즉 이 spec은 **현재 보상 메카닉을 유지한 채 phase와 화면만 추가**하는 bounded 작업이다.

## 4. 제안 구현안: `오버런 경계 이벤트 페이지`

### 4.1 phase 이름

권장 phase 명칭:

- `overrunEvent`

현재 `phase` 체계가 `library / battle / shop / reward / ending`이라면, 이번 추가는 여기에 `overrunEvent`를 1개 더 넣는 방향을 우선안으로 둔다.

이유:

- `ending` 내부 서브상태로 밀어 넣으면 UI 조건문이 다시 비대해진다.
- 새 페이지 구조라는 점을 코드/문서/UI 모두에서 같은 이름으로 부를 수 있다.
- 이후 짧은 오버런 이벤트를 1종 더 붙일 때도 확장 포인트가 명확하다.

### 4.2 진입 조건

아래 조건을 모두 만족할 때 진입한다.

1. `ending.victory === true`
2. `ending.canOverrun === true`
3. 플레이어가 `오버런 계속`을 눌렀다.

### 4.3 미진입 조건

아래 중 하나면 `overrunEvent`를 생략하고 기존 흐름을 유지한다.

- 패배 엔딩
- 오버런 불가 엔딩
- 세이브 호환 복구 중 `ending` 정보가 불완전한 경우

단, **현재 spec 우선안은 가능한 한 생략하지 않고 항상 진입**하는 것이다.  
선택 전리품이 없더라도 `빈손으로 다음 권을 넘긴다`는 fallback 페이지를 보여 주는 편이 UX적으로 더 낫다.

### 4.4 페이지 1회성 규칙

- 세그먼트당 1회만 표시
- `오버런 계속` 직후 표시
- 확인 버튼 후 바로 다음 세그먼트 첫 페이지(`segmentPage = 1`)로 이동
- 같은 세그먼트 안에서 다시 열리지 않음

## 5. 플레이어 플로우

### 5.1 전리품을 선택한 경우

1. 책 클리어 엔딩에서 `오버런 전리품 보관함`에서 1개 선택
2. 플레이어가 `오버런 계속` 선택
3. `overrunEvent` 페이지 진입
4. 중앙에 선택한 전리품 1장을 크게 표시
5. 상단에 `다음 권 서문` 성격의 짧은 문장 표시
6. 하단에 `이 전리품을 챙기고 다음 권으로` 단일 CTA 표시
7. CTA 클릭 시 기존 로직대로 전리품 즉시 적용 후 다음 세그먼트 첫 페이지 진입

### 5.2 선택 없이 넘어간 경우

1. 책 클리어 엔딩에서 선택하지 않고 `오버런 계속`
2. `overrunEvent` 페이지 진입
3. 중앙에는 `빈 서가` 또는 `전리품 없음` 상태 카드 표시
4. 상단 문구는 `이번엔 아무 잔해도 챙기지 않고 넘긴다` 계열로 처리
5. CTA는 `가볍게 다음 권을 연다`
6. 추가 효과 없이 다음 세그먼트 첫 페이지 진입

핵심은 **skip도 페이지 경험으로 처리한다**는 점이다.

## 6. 상태 / 데이터 명세

### 6.1 새 상태 오브젝트

권장 런타임 필드:

```json
{
  "phase": "overrunEvent",
  "overrunEvent": {
    "sourceType": "themeTrophy | enemyTrophy | none",
    "rewardId": "string | null",
    "rewardTitle": "string | null",
    "rewardDescription": "string | null",
    "theme": "slime | goblin | lich | null",
    "fromTemplateId": "number | null",
    "resolved": false
  }
}
```

### 6.2 필드 의미

- `sourceType`
  - `themeTrophy`: 책 테마 전리품에서 온 선택
  - `enemyTrophy`: `기록 골렘` / `붕락 정령` 등 적 전리품에서 온 선택
  - `none`: 선택 없이 넘어간 경우
- `rewardId`
  - 기존 reward 정의 재사용
- `rewardTitle`, `rewardDescription`
  - 렌더 단계에서 reward lookup을 다시 하지 않아도 되도록 snapshot 성격으로 저장 가능
- `theme`
  - 헤더/색상/문구 톤 분기용
- `fromTemplateId`
  - 적 전리품일 때만 사용
- `resolved`
  - CTA를 눌러 실제 적용/이동이 끝났는지 표시

### 6.3 데이터 원칙

- 이번 spec은 **새 reward definition을 만들지 않는다.**
- 기존 `ending.overrunDraft.selectedRewardId`와 reward metadata를 사용해 `overrunEvent` snapshot만 만든다.
- `none`일 때도 `overrunEvent` 오브젝트는 만든다. 그래야 렌더 경로와 세이브 복구 경로가 단순해진다.

### 6.4 적용 시점

현재 구현이 `오버런 진입 직전에 즉시 적용`이라면, 그 타이밍을 아래처럼 고정한다.

- `오버런 계속` 클릭 시 즉시 적용하지 않음
- `overrunEvent`의 CTA 클릭 시 적용
- 적용 후 다음 세그먼트 준비 / 전투 진입

이렇게 해야 페이지가 진짜 `선택을 들고 넘어가는 장면`이 된다.

### 6.5 세이브 호환 원칙

구세이브나 중간 저장 호환을 위해 아래 보정을 둔다.

- `phase === 'overrunEvent'`인데 `overrunEvent`가 없으면 fallback 생성
- `rewardId` lookup이 실패하면 `sourceType = none`으로 강등
- `resolved === true` 상태로 로드되면 다음 페이지 진입 로직을 재실행하지 않고 바로 battle/shop 전개 상태로 복구

## 7. 프로그래머 handoff

### 7.1 구현 목표

프로그래머에게 필요한 일은 `새 보상 시스템`이 아니라 아래 4개다.

1. `overrunEvent` phase 추가
2. `ending -> overrunEvent -> next segment` 흐름 추가
3. 기존 선택 보상을 CTA 시점에 적용하도록 타이밍 이동
4. 스모크 테스트와 세이브 정규화 추가

### 7.2 권장 구현 단위

#### A. game-engine

필요 함수 예시:

- `createOverrunEventState(state)`
- `resolveOverrunEvent(state)`
- `normalizeOverrunEventState(state)`

책임 분리:

- `createOverrunEventState`
  - ending snapshot을 읽어 `overrunEvent` payload 생성
- `resolveOverrunEvent`
  - 선택한 전리품 적용
  - 다음 세그먼트 상태 준비
  - `phase = next page target`로 이동
- `normalizeOverrunEventState`
  - 세이브 복구 / fallback 처리

#### B. app.js

필요 렌더/액션 예시:

- `renderOverrunEvent()`
- `handleConfirmOverrunEvent()`

책임 분리:

- 중앙 카드 / 헤더 / 설명 / CTA 렌더
- skip 상태의 빈 카드 렌더
- 클릭 후 resolve 호출

#### C. styles.css

필요 스타일 범위:

- 기존 reward card 재활용 가능
- 단, 중앙 집중형 배치와 헤더 장식 정도는 별도 클래스 추가 권장

### 7.3 acceptance criteria

1. 책 클리어 후 `오버런 계속`을 누르면 battle로 바로 가지 않고 `overrunEvent`를 1회 거친다.
2. 전리품을 골랐다면 해당 카드가 크게 다시 보인다.
3. 선택하지 않았다면 빈손 fallback 카드가 보인다.
4. CTA 클릭 시 기존 기대와 동일한 효과가 적용된다.
5. 다음 세그먼트 첫 페이지 진입 전 중복 적용되지 않는다.
6. 세이브/로드 후에도 event state가 깨지지 않는다.

### 7.4 smoke 기준

최소 smoke 케이스:

1. `selectedRewardId`가 있는 승리 엔딩 -> `overrunEvent` 생성
2. `sourceType=themeTrophy` 렌더 payload 정상 생성
3. `sourceType=enemyTrophy` 렌더 payload 정상 생성
4. `selectedRewardId=null` -> `sourceType=none` fallback 생성
5. CTA 클릭 시 reward 1회만 적용
6. 세그먼트 첫 페이지 진입 후 `overrunEvent` 재진입 없음
7. 구세이브 복구 시 invalid rewardId -> `none` fallback 강등

이번 작업은 docs-first이므로 구현 시작 시 smoke부터 먼저 고정하는 쪽이 안전하다.

## 8. UI handoff

### 8.1 화면 목적

UI의 목표는 선택지를 더 주는 것이 아니다.  
**방금 고른 전리품 1개를 엔딩 정산 문맥에서 떼어내 다음 권 첫 장 문맥으로 다시 보여 주는 것**이 핵심이다.

### 8.2 화면 구조

권장 3단 구조:

1. 상단: 짧은 전환 헤더
2. 중앙: 전리품 카드 1장 중심 영역
3. 하단: CTA 1개 + 보조 설명 1줄

### 8.3 헤더 카피 구조

헤더는 시스템 설명보다 **전환 감각**이 먼저 와야 한다.

예시 톤:

- `다음 권의 첫 장을 펼친다`
- `방금 건진 잔해를 챙긴다`
- `주운 것을 들고 다음 기록으로 넘어간다`

주의:

- 여기서 기계적으로 `reward applied` 같은 문구를 전면에 두지 않는다.
- 기능 설명은 카드 설명과 하단 보조 문구에서 처리한다.

### 8.4 중앙 카드 규칙

- 기존 reward card를 기반으로 하되 크기를 1.2~1.4배 확대
- 카드 좌상단에 출처 배지 추가
  - `책 전리품`
  - `적 전리품`
  - `빈손`
- 카드 하단에는 기존 효과 요약 유지

### 8.5 하단 CTA 규칙

선택 전리품 있음:

- 기본 버튼: `이 전리품을 챙기고 다음 권으로`

선택 전리품 없음:

- 기본 버튼: `가볍게 다음 권을 연다`

보조 문구:

- `확인 후 다음 세그먼트 첫 페이지로 이동합니다.`

### 8.6 정보 우선순위

UI는 아래 순서로 보이게 한다.

1. 지금은 `세그먼트 경계`다.
2. 내가 챙길 것은 이것이다.
3. 이 버튼을 누르면 다음 권으로 넘어간다.

즉, 숫자보다 **문맥 -> 대상 -> 행동** 순서가 중요하다.

### 8.7 skip 상태 디자인

skip 상태를 빈 화면으로 두면 안 된다.

최소 요구:

- 플레이스홀더 카드 1장
- `챙긴 전리품 없음` 라벨
- 다음 권으로 넘어가는 CTA

이 fallback이 있어야 플레이어가 `선택하지 않은 것도 게임이 이해했다`는 확신을 가진다.

## 9. 아트 handoff

### 9.1 이번 작업의 아트 범위

이번 단계는 대형 일러스트 제작이 아니라 **기존 카드/패널 자산 위에 덧씌우는 소형 패키지**가 적절하다.

필수 자산:

1. `오버런 이벤트` 헤더용 패널 배경 1종
2. 출처 배지 3종
   - 책 전리품
   - 적 전리품
   - 빈손
3. 중앙 카드 강조용 backplate / glow 1세트
4. 테마 미세 변주용 accent 색 규칙
   - slime: 안정/회복 계열 녹청
   - goblin: 노획/경제 계열 황동
   - lich: 금서/속성 계열 자주/청보라

### 9.2 권장 표현

- 큰 새 일러스트보다 `책상`, `제본대`, `열린 책`, `올려둔 전리품` 같은 UI 프레임 암시가 더 중요하다.
- 즉시 구현 우선안은 `2D full art`가 아니라 `패널/배지/광원` 패키지다.
- 카드 자체 일러스트가 부족하더라도 backplate와 헤더가 있으면 장면 전환감은 충분히 살 수 있다.

### 9.3 피해야 할 것

- 전리품마다 새 삽화 요구
- theme별 완전 별도 레이아웃 요구
- 애니메이션 의존도가 높은 연출

이번 단계는 **빠르게 붙일 수 있는 handoff-ready UI art 패키지**가 맞다.

## 10. 카피 가이드

### 10.1 공통 원칙

- 짧고 명확해야 한다.
- `설명`보다 `넘어간다`는 감각이 먼저 와야 한다.
- 책/적의 잔향을 느끼게 하되 장황하면 안 된다.

### 10.2 출처별 보조 카피 예시

#### 책 전리품

- `이번 책의 결이 아직 손에 남아 있다.`
- `남은 잔향을 다음 장의 준비물로 삼는다.`

#### 적 전리품

- `막 쓰러뜨린 위협의 잔해를 챙긴다.`
- `방금 넘긴 적의 결을 다음 권 첫 장에 들고 간다.`

#### 빈손

- `이번엔 남긴 것 없이 가볍게 넘긴다.`
- `손은 비었지만 다음 장은 열린다.`

## 11. 구현 우선순위와 out-of-scope 명시

이번 이벤트 페이지 구현에서 가장 중요한 것은 **boundedness**다.

반드시 지킬 것:

1. 카드 1장 중심
2. CTA 1개 중심
3. 추가 보상/재선택 없음
4. 기존 reward 정의 재사용
5. smoke로 복구 경로까지 먼저 고정

이번 단계에서 하지 말 것:

- 전리품 변환
- 전리품 강화
- 두 번째 선택지 제공
- 오버런 경로 분기
- 별도 맵 노드 생성

이걸 어기면 구현이 갑자기 시스템 확장으로 변한다.

---

## 12. 리스크와 후속 과제

### 현재 리스크

1. `ending`과 `overrunEvent`의 책임 분리가 어설프면 UI 조건문이 더 꼬일 수 있다.
2. 선택 보상 적용 시점을 늦추면서 중복 적용 버그가 날 수 있다.
3. skip 상태를 허술하게 만들면 `굳이 왜 이 페이지가 있지?`라는 인상을 줄 수 있다.

### 현재 대응

- `phase`를 명시적으로 분리한다.
- `resolved` 플래그와 smoke로 중복 적용을 막는다.
- skip 상태도 동일한 레이아웃과 CTA를 유지한다.

### 후속 과제

이 페이지가 안정적으로 붙고 나면 그다음 후보는 둘 중 하나다.

1. `보스형 골렘`처럼 전리품 서사를 더 강하게 만드는 상위 위협 1종 추가
2. `overrunEvent`를 재사용하는 짧은 오버런 이벤트 바리에이션 1종 추가

단, 둘 다 이번 단계가 안정된 뒤다.

---

## 13. 한 줄 handoff 요약

- **프로그래머:** `overrunEvent` phase를 추가하고, 기존 오버런 선택 보상 적용 시점을 이 페이지 CTA로 옮긴다.
- **UI 오너:** 카드 1장 강조형 세그먼트 경계 화면을 설계하고, `문맥 -> 대상 -> 행동` 순서로 정보를 배치한다.
- **아트 오너:** 새 풀 일러스트가 아니라 헤더 패널 / 출처 배지 / 카드 강조 backplate 중심의 소형 패키지를 준비한다.

이 문서의 목표는 `무엇을 만들지`보다 `어디까지 만들지`를 명확하게 고정하는 데 있다.
